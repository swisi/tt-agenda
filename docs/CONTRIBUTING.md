# Contribution Guide (v2)

## Arbeitsweise

- Kleine, fokussierte Pull Requests
- Tests für neue Use Cases
- Keine Features direkt im Legacy-Archiv

## Branching

- `main`: stabil
- `feature/*`: neue Entwicklung
- `refactor/*`: strukturelle Umbauten

## Qualitätsregeln

- Linting + Tests müssen grün sein
- Öffentliche Interfaces dokumentieren
- Breaking Changes explizit kennzeichnen
